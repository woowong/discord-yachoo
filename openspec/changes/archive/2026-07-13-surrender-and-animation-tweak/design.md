## Context

현재 Yacht 다이스 게임에서 유저가 중간에 게임을 포기하고 조기 종료할 수 있는 공식적인 장치가 부재하여 무의미한 대기 시간이 발생합니다.
또한 비동기 다이스 롤링 연출 대기 시간(1.4초)이 흐름을 지연시키는 느낌이 있어, 이를 1.2초로 튜닝하여 사용자 경험을 신속하게 개선하고자 합니다.

## Goals / Non-Goals

**Goals:**
- 1인 플레이(single) 및 2인 플레이(multi) 모드 모두에서 작동하는 기권 처리 순수 도메인 함수 설계
- 턴 소유권 여부와 관계없이 해당 게임의 참가자 플레이어라면 누구나 언제든지 기권 버튼을 통해 기권할 수 있도록 처리
- 기권 발생 시 ELO 변동과 매치 결과 DB 적재가 정상적으로 승리자/패배자 매핑에 맞게 처리되도록 구현
- 중복되는 게임 종료 로직을 단일 공통 헬퍼 Effect(`handleGameEnd`)로 통합
- 연출 지연 속도를 1.4초에서 1.2초로 개선
- 본인 자신에게 `/challenge` 대전을 신청하는 비정상적인 매칭 차단

**Non-Goals:**
- D1 데이터베이스의 `matches` 테이블에 신규 컬럼을 추가하는 등의 스키마 마이그레이션 (기존 `winner_id` 컬럼으로 ELO와 승패 반영이 충분히 가능하며, `active_games`는 JSON으로 통째로 영속화되므로 스키마 변경 최소화)
- 기권 확인을 위한 2단계 확인 창 UI 도입 (1단계 단일 버튼 처리를 원칙으로 함)

## Decisions

### 1. `GameState` 내 기권 플레이어 추적 필드 추가
- **선택**: `GameState` 인터페이스에 선택적 필드로 `surrenderedPlayerId?: string`를 추가합니다.
- **이유**: `active_games` 임시 저장용 테이블의 `state` 컬럼은 JSON 문자열로 저장되므로, 굳이 D1 스키마 마이그레이션을 하지 않아도 필드 확장이 자유롭습니다.

### 2. 턴 소유권과 독립적인 기권 처리 및 권한 제어
- **선택**: `src/index.ts` 내 턴 소유자 검증(`Validate turn`) 로직 이전 단계에서 기권(custom_id: `"surrender"`) 인터랙션을 선제 파싱하여 처리합니다. 단, 해당 게임 참가자(`isPlayerInGame`)가 아닌 사용자가 누를 경우 에러 메시지를 반환합니다.
- **이유**: 상대방이 잠수를 타거나 턴을 무기한 붙잡고 있을 때에도 기권하여 매치를 종료할 수 있도록 턴 독립적으로 동작해야 합니다.

### 3. 게임 종료 처리 로직의 공통화 리팩토링
- **선택**: `selectCategory`에 의한 자연 종료와 `surrender`에 의한 강제 종료 모두에서 동일한 매치 저장, Elo 갱신, 알림 전송 로직이 실행될 수 있도록 `handleGameEnd` 헬퍼 Effect를 작성합니다.
- **이유**: ELO 변동 계산, D1 데이터베이스 적재, 디스코드 알림 발송 등 복잡한 비즈니스 사이드 이펙트의 중복 코드를 제거하여 유지 보수성을 높입니다.

### 4. 자기 자신에게 대전 신청 차단 유효성 검사 추가
- **선택**: `/challenge` Slash Command 수신 시, 대전 상대인 `opponentId`가 입력되었고 그 값이 명령어 요청 사용자인 `interaction.user.id`와 동일하다면, 매칭 처리를 즉각 취소하고 Ephemeral 에러 응답(type: 4, flags: 64)을 반환합니다.
- **이유**: 게임 모델의 일관성을 저해하는 부적절한 대전 매칭을 원천 차단하기 위함입니다.

## Risks / Trade-offs

- **실수로 인한 기권 발생 가능성**: 단일 클릭으로 즉시 게임이 기권 처리되므로 실수로 누를 위험이 존재합니다.
  - *Mitigation*: Discord 버튼 스타일 중 Danger(Red, `style: 4`)를 적용하여 시각적 주의를 환기하고, 주사위가 실제로 굴러가는 연출 상황(`serializeRolling`)에서는 기권 버튼을 비활성화하여 오작동 가능성을 경감합니다.
