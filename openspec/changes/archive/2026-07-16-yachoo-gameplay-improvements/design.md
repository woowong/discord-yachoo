## Context

현재 디스코드 야추 게임 봇 서비스의 사용성(UX) 및 데이터베이스 효율성 개선을 목표로 하고 있습니다. 주요 핵심 맥락은 다음과 같습니다:
- **중복 대전**: 유저 간 여러 게임이 동시에 개설되어 상태 관리가 어긋나는 오류가 발생합니다.
- **DB 스토리지**: `active_games` 테이블에 완료된 게임(Finished)이 삭제되지 않고 계속 남아있어 D1 스토리지 공간이 비효율적으로 낭비됩니다.
- **기권 프로세스**: 사용자가 "기권" 버튼을 실수로 눌렀을 때 취소 기회가 없어 불이익이 발생합니다.
- **디스코드 UI 가독성**: 모바일 기기 등 저해상도 화면에서 주사위 홀드 버튼의 글자 수(예: `[1] Held`) 변경에 따라 가로 너비가 유동적으로 변하여 레이아웃이 줄바꿈되거나 깨집니다.
- **사용자 흥미 요소**: 야추(Yacht) 족보 완성 또는 안타까운 야추 포기 시점 등의 극적 순간에 디스코드 채널에 고유한 톤앤매너로 알림과 박제를 진행하여 커뮤니티적 가치를 올립니다.

## Goals / Non-Goals

**Goals:**
- 동일 사용자 조합 간의 활성 게임(InProgress) 개수를 최대 1개로 제한.
- 중복 대전 시 기존 게임의 Discord 메시지로 바로갈 수 있는 포워딩 링크(URL) 제공.
- 완료된 게임은 `active_games` 테이블에서 삭제(DELETE)하도록 트랜잭션 수명 주기 개선 및 기존 데이터 정리 마이그레이션.
- 기권 클릭 시 에페메랄(비공개) 메시지 창으로 실제 기권 의사 재확인.
- 주사위 홀드 버튼 너비를 일반 상태(`[1]`)와 Held 상태(`🔒`)의 글자 너비를 매칭하여 UI 고정.
- 야추 달성, 야추 소모 후 야추 획득, 낮은 점수, 연속 0점, 최종 꼴찌 발생 시 특정 커뮤니티 밈 어조의 축하/조롱 메시지를 발송하도록 룰 추가.

**Non-Goals:**
- 게임 룰(야추 족보 점수 계산식) 자체를 변경하지 않음.
- 디스코드 슬래시 커맨드 스키마 자체의 대대적인 변경(예: `/challenge` 옵션 구성 자체는 변경 없음).

## Decisions

### 1. 중복 대전 조회 및 메시지 포워딩
- **결정**: `GameRepository`에 `findActiveGameByPlayers(player1Id: string, player2Id: string)` 인터페이스를 신설하여 대전 생성 전 중복을 검사합니다.
- **메시지 링크 저장**: 대전이 처음 생성되면, 최초 Discord Slash Command에 대한 응답 메시지 ID가 확정되지 않습니다. 따라서 `/challenge` 처리의 백그라운드 태스크나 첫 멘션(Mention) 저장 시점에 `@original` interaction message ID를 Fetch하여 `initialMessageId`를 `GameState` 내에 업데이트하고 DB에 저장합니다.
- **대안 고려**:
  - *대안 A*: 메시지 링크 없이 게임 ID만 출력. (사용자가 직접 스크롤해서 찾아야 하므로 불편하여 기각)
  - *대안 B (채택)*: Discord Interaction Token을 사용하여 `@original` 메시지 세부정보를 GET/PATCH하고, 여기서 획득한 메시지 ID로 `https://discord.com/channels/{guild_id}/{channel_id}/{message_id}` 링크를 빌드하여 제공.

### 2. active_games 데이터의 Life Cycle 및 마이그레이션
- **결정**: 게임이 `Finished` 상태가 되면 `active_games` 테이블에서 `DELETE` 쿼리를 실행합니다.
- **하위 호환성 및 정리**:
  - `active_games`에 이미 존재하던 Finished 게임들을 삭제하기 위해 마이그레이션 스크립트(`migrations/0007_cleanup_finished_active_games.sql`)를 제공해 패치 시점에 D1 DB를 한 번 청소합니다.
  - 혹시 모를 누락을 대비해, 게임 조회 시에는 여전히 `json_extract(state, '$.status') = 'InProgress'` 조건을 안전장치로 유지합니다.

### 3. 기권 에페메랄 확인 흐름
- **결정**: 기권 버튼(`custom_id: surrender`) 클릭 시, 즉시 기권을 실행하지 않고 `type: 4`(ChannelMessageWithSource)에 `flags: 64`(Ephemeral) 플래그를 붙여 유저 개인에게만 보이는 메시지를 반환합니다.
- **최종 기권**: 에페메랄 메시지 내의 확인 버튼(`custom_id: confirm_surrender`)을 클릭해야 실제 기권 비즈니스 로직(`surrenderGame`)을 실행하고 게임 상태를 업데이트 및 메인 메시지를 갱신합니다.

### 4. 고정 폭 주사위 버튼 라벨 설계
- **결정**:
  - 일반 상태: `[1]`, `[2]`, `[3]`, `[4]`, `[5]` (각 3글자 너비)
  - Held 상태: `🔒` 이모지 문자. (디스코드 클라이언트에서 3글자 정도의 가로 너비와 물리적으로 거의 동등하게 렌더링됨)
  - 이로써 버튼의 가로 너비 변화가 없어 클라이언트 단의 UI 찌그러짐이나 줄바꿈이 완벽히 예방됩니다.

### 5. 야추/Poor-play 어그로 및 놀림 메시지 송신
- **결정**: `src/index.ts` 내의 카테고리 선택 및 게임 종료 트랜잭션 직후, 상태 변화를 감지하여 Discord Webhook API(또는 `DiscordApiService`)를 호출해 별도의 텍스트 메시지를 채널에 발송합니다.
- **판별 규칙**:
  - **Yacht 축하**: 기록 점수가 50점이고 카테고리가 `Yacht`일 때.
  - **야추 헛박제(조롱)**: 5개 주사위 눈이 일치(Yacht 조건 충족)하나, 이미 `scoreBoard.Yacht` 슬롯이 채워진 채로 다른 카테고리에 기입할 때.
  - **낮은 점수**: Aces에 0~1점 기록 또는 Choice에 5점 미만 기록 시.
  - **연속 0점**: 이번 턴에 0점을 기록했는데, `turnHistory` 상 바로 직전 턴에도 0점을 기록한 경우 (2턴 연속 이상).
  - **최종 꼴찌**: 게임 Finished 상태 돌입 후, 멀티플레이 모드에서 더 낮은 점수를 획득한 플레이어 확정 시.

## Risks / Trade-offs

- **[Risk] 기권 확인 에페메랄 버튼 만료**
  - *설명*: Discord의 interaction token은 15분 후 만료되므로, 기권 확인 창을 띄워두고 오랜 시간 방치한 뒤 "확인"을 누르면 실패할 수 있습니다.
  - *완화*: 기권 확인 클릭 시 token 만료 에러가 나면 유저에게 만료되었음을 알리고 메인 게임 보드의 기권 버튼을 다시 이용하도록 가이드합니다.
- **[Risk] 마이그레이션 시 데이터 유실**
  - *설명*: active_games 테이블에서 Finished 게임을 DELETE할 때 진행 중인 게임이 잘못 삭제될 가능성.
  - *완화*: D1 DB의 `json_extract` 파싱 조건을 엄격히 검사하고 로컬 테스트 환경(`vitest`)에서 검증 후 안전하게 마이그레이션을 적용합니다.
