## Context

디스코드 야추 게임(discord-yachoo)은 Cloudflare Workers 위에서 동작하는 서버리스 봇으로, Effect.ts 기반의 함수형 아키텍처를 따른다. 현재 3개의 계층(Domain/Persistence/Presentation)이 명확히 분리되어 있으며, D1 데이터베이스에 플레이어 통계와 매치 기록을 저장한다.

실사용자 피드백으로 5가지 개선이 요구되었다:
1. 모바일 디스코드에서 스코어보드가 깨지는 문제 (코드블록 가로 39자 → 모바일 한계 30~33자 초과)
2. 보너스 기준점수(63)의 진행 상황 불투명
3. 전체 홀드 상태에서 무의미한 롤 허용
4. 게임 종료 시 알림 부재 (UpdateMessage는 디스코드 알림을 발생시키지 않음)
5. 경쟁 동기부여를 위한 ELO 레이팅 시스템 요구

## Goals / Non-Goals

**Goals:**
- 모바일 디스코드에서 스코어보드가 줄바꿈 없이 정상 표시되도록 가로 27자 이내로 축소
- 상단 섹션 합산을 `현재/63` 진행률 형태로 표기
- 5개 주사위 전체 홀드 시 롤 불가능하도록 UI/도메인 이중 방어
- 게임 종료 시 별도 메시지로 승자 @멘션 알림 전송 (reply 연결)
- Elo 레이팅 계산·저장·노출 (멀티 모드 전용)

**Non-Goals:**
- 스코어보드를 Discord Embed 필드 기반으로 전환하는 것 (코드블록 방식 유지)
- 솔로 모드에 Elo 적용
- Elo 시즌/리셋 시스템
- 3인 이상 멀티플레이어 Elo 계산

## Decisions

### Decision 1: 스코어보드 가로 27자 제한

**선택**: 닉네임 4글자 제한 + 카테고리 레이블 단축 + 점수 5자 폭

```
구조: label(10) + " |" + score(5) + " |" + score(5) = 27자
```

**대안 검토**:
- Embed 필드 기반 렌더링 → 카테고리 14개를 필드로 표현하면 메시지가 과도하게 길어지고, 플레이어별 열 정렬이 불가능
- 이미지 렌더링 → Workers 환경에서 Canvas API 미지원, 외부 서비스 의존도 증가

**결정 근거**: 코드블록(`\`\`\``) 방식이 모노스페이스 폰트로 열 정렬이 보장되며, 기존 구조를 최소한으로 변경하면서 모바일 호환성을 확보할 수 있다.

### Decision 2: 보너스 진행률 `현재/63` 표기

**선택**: Subtotal 행의 값을 `0/63` 형태로 표기

**결정 근거**: 숫자만 표시하면 63이라는 기준을 모르는 사용자가 보너스 달성 여부를 판단할 수 없다. `현재/목표` 형태는 직관적이며, 게이지바 같은 시각 요소 없이도 충분히 정보를 전달한다.

### Decision 3: 전체 홀드 방지 — UI + 도메인 이중 방어

**선택**: Presentation 레이어에서 `disabled: true`, Domain 레이어에서 `AllDiceHeldError`

**대안 검토**:
- UI만 방어 → API를 직접 호출하는 경우(드물지만) 방어 불가
- 도메인만 방어 → 사용자에게 에러 메시지가 표시되는 나쁜 UX

**결정 근거**: 아키텍처 원칙(계층 간 격리)에 따라 비즈니스 규칙은 도메인에, UX 개선은 프레젠테이션에 각각 배치하는 이중 방어가 적절하다.

### Decision 4: 승자 알림 — 별도 메시지 + reply 연결

**선택**: `DiscordApiService`에 `sendGameEndMessage` 메서드를 추가하고, 게임 종료 시 원본 게임 메시지에 reply로 별도 알림 전송

**대안 검토**:
- Embed의 `content` 필드에 멘션 추가 → `type: 7` (UpdateMessage)는 디스코드 알림(notification)을 발생시키지 않으므로 실효성 없음

**결정 근거**: 새로운 메시지만이 디스코드 알림을 트리거할 수 있다. `message_reference`로 원본 게임 메시지에 연결하면 컨텍스트도 유지된다.

### Decision 5: Elo 순수 함수 분리 (src/domain/elo.ts)

**선택**: Elo 계산 로직을 `src/domain/elo.ts`에 순수 함수로 분리. 기본값 1000, K-factor 32, 최저값 100.

**결정 근거**: 아키텍처 원칙에 따라 Core Game Engine은 순수 함수로만 구성. Elo 계산은 외부 의존성이 없는 수학 연산이므로 도메인 레이어에 자연스럽게 배치된다. K=32는 캐주얼 게임 특성(빠른 실력 반영)에 적합하며, 최저값 100은 극단적인 연패에서도 의미 있는 레이팅을 유지시킨다.

### Decision 6: Elo 저장 — 기존 테이블 확장

**선택**: `players`와 `guild_player_stats` 테이블에 `elo INTEGER NOT NULL DEFAULT 1000` 칼럼 추가

**대안 검토**:
- 별도 `elo_ratings` 테이블 → 조인 비용 증가, 리더보드 쿼리 복잡화
- `matches` 테이블에 Elo 변동 기록 → 히스토리 추적에는 유용하나 현재 스코프 초과

**결정 근거**: 기존 `guild_player_stats` 테이블이 이미 길드별 통계를 관리하고 있으므로, elo 칼럼을 추가하는 것이 가장 자연스럽고 쿼리 효율적이다.

## Risks / Trade-offs

| Risk | Mitigation |
|------|------------|
| 닉네임 4글자 제한으로 사용자 식별이 어려울 수 있음 | 디스코드에서는 멘션(@)으로 사용자를 확인할 수 있으므로 스코어보드의 짧은 닉네임은 보조 식별용 |
| 기존 게임 진행 중 스코어보드 포맷 변경 시 혼란 | 포맷 변경은 다음 응답부터 적용되므로 진행 중인 게임도 자연스럽게 전환됨 (상태 변경 없이 렌더링만 변경) |
| D1 마이그레이션 시 기존 데이터에 elo=1000 일괄 적용 | ALTER TABLE의 DEFAULT 1000이 기존 행에도 적용됨. 모든 플레이어가 동일 선에서 시작하므로 공정 |
| Elo 최저값 100 제한으로 실제 실력 차이 표현이 왜곡될 수 있음 | 100까지 내려가려면 극단적 연패가 필요하므로 현실적으로 영향 미미 |
